ELECTION DATA · DESIGN SYSTEM v1.0

Warm-neutral civic interface system

Quiet surfaces. Clear hierarchy. Strong operational focus.

This document now represents the design system itself, not just isolated tokens. The goal is a restrained interface language for election data products, dashboards, briefings, and review tools: warm-neutral surfaces, charcoal typography, deliberate accent usage, and repeatable interaction patterns that hold up under dense information.

Accessible by default
Default surface direction Warm beige Calmer than stark white, still structured enough for data-heavy layouts.
Text behavior Charcoal ink Stable across all light themes so legibility does not drift with palette changes.
Interaction rule One loud action Primary emphasis is rare; everything else should support the decision path.
Principle 01 Neutral before expressive

Most UI area belongs to surface, text, border, and spacing discipline, not brand color.

Principle 02 Meaning before taste

Success, warning, and error should communicate state without becoming the text color.

Principle 03 Reuse shells, not just tokens

Dashboards, queues, forms, and empty states should feel like one product family.

Principle 04 Complexity is staged

Dense tables and review flows are fine, but the page should reveal them with hierarchy.

Foundation colors

These are the foundational color families that carry the system. Neutrals and surfaces do most of the work; accent and semantic hues are intentionally controlled.

Cool-neutral gray, not warm-cream — reads as institutional/data-grade rather than lifestyle-brand. This is the majority of every screen; the accent should feel rare by comparison.

Action + semantic color

Deliberately not saffron, green, or party-adjacent blue — this is an elections tool, so the accent needs to read as neutral infrastructure, not political signal. Semantic colors are muted, not flag-bright, for the same reason.

Charts + categorical cues

Colorblind-safe by construction (Okabe-Ito derived) — this row isn't a style choice, it's the one place aesthetics take a back seat to accessibility science. Use in this order; don't reach for red/green as a pair for meaning. Texture belongs on supporting component surfaces like cards, labels, and legend keys, not inside dense charts.

Texture + motion study

Use this area to audition accent directions without editing code and to see where textured categorical treatments help. The default indigo system is still the best baseline for civic data UI; the others are comparison studies.
